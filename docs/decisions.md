# Design Decisions（設計判断の記録 / ADR）

「なぜそうしたか」を未来の自分が思い出すための記録。
各決定は、状況・決定・理由・代替案で構成する。

---

## ADR-001: 単一ユーザー前提で作る

- **状況**: ラズパイ上で動かす自分専用ツール。家庭内 LAN で使う。
- **決定**: ログイン認証を入れない。全テーブルに `user_id` を持たせない。ただしポートフォリオは `portfolio_id` で複数持てる器にしておく。
- **理由**: 利用者が自分 1 人なので認証・セキュリティ・公開デプロイが不要になり、全体が大幅にシンプルになる。`portfolio_id` だけ残すことで、将来の複数ユーザー化を完全には殺さない。
- **代替案**: 複数ユーザー対応（認証・HTTPS・公開デプロイ）→ YAGNI。今は過剰。

## ADR-002: データベースは SQLite（WAL モード）

- **状況**: 扱うのは日足データ。4000 銘柄 × 250 営業日 × 10 年 ≈ 1000 万行程度。tick データは扱わない。
- **決定**: SQLite を採用し、WAL モードを有効にする。
- **理由**: 単一ユーザー・読み中心なら数千万行は SQLite で十分。WAL により夜間バッチの書き込みと画面の読み取りが衝突しにくい。別プロセス（DB サーバー）を立てずメモリを節約できる。
- **代替案**: PostgreSQL → 「Next と Python が同時に大量書き込み」または「tick 級データ」になったら再検討。今回は該当しない。

## ADR-003: ラズパイ 4B でローカル完結、クラウド不使用

- **状況**: 非リアルタイム（毎晩・毎月の定期処理）用途。自動売買のミリ秒監視は不要。
- **決定**: 計算・DB・配信をすべてラズパイ 4B（8GB 推奨）に集約。Supabase / Vercel / AWS は使わない。
- **理由**:
  - Supabase: 認証・リアルタイムが売りだが単一ユーザーには無駄。大量データのクラウド往復が遅く無料枠も逼迫。
  - Vercel: Python の常駐バッチ・重い計算を動かせない。フロントだけクラウドにすると裏のラズパイを外部公開する羽目になり複雑化。
  - AWS 無料枠: 12 か月限定かつメモリ 1GB でラズパイより非力、Pandas/ML が OOM しやすい。
  - ラズパイは無料・8GB・常時起動・データもローカルで完結し、本用途では最も合理的。
- **代替案**: 全クラウド構成 → 将来 ML 学習を頻繁に回す等で必要になったら再検討。

## ADR-004: フロント=Next.js / バック=FastAPI（スタック A）

- **状況**: 利用者の既存スキルは Laravel / Vue。React / Next / Python は未経験。
- **決定**: README 通り Next.js（App Router）+ FastAPI を採用。
- **理由**:
  - バックエンドが Python なのは**交渉の余地なく確定**。PyPortfolioOpt / TA-Lib / LightGBM は PHP に存在せず、数理・ML がこの製品の心臓だから。
  - フロントは Vue 寄せ（Nuxt）や Streamlit も候補だったが、**利用者が「React/Next を学ぶこと自体を目的に含めて」自ら選択**したため Next.js を採用。
- **代替案**: Nuxt（Vue 既知）/ Streamlit（Python 1 本・最速）→ 学習目的を優先して不採用。

## ADR-005: DB に触れるのは FastAPI のみ（Next は REST 経由）

- **状況**: README 初稿は「Next が Prisma で DB 連携」「FastAPI が SQLite 接続」と、2 プロセスが同じ SQLite を触る構図だった。
- **決定**: DB に触れるのは FastAPI だけ。Next.js は DB を直接触らず、すべて FastAPI の REST API 経由で取得する。**Prisma は採用しない。**
- **理由**:
  - SQLite は「書き込みは同時に 1 つだけ」。書き手を Python 1 つに限定すればロック競合が原理的に起きない。
  - スキーマ定義が Python 側 1 箇所に集約され、二重管理による事故を防ぐ。
  - 責務が「Next = UI / FastAPI = データ・計算」と明確に分かれ、学習時の混乱も減る。
- **代替案**: README 通りの役割分担（Next+Prisma と Python が DB 共有）→ 書き込み競合・スキーマ二重管理の罠のため不採用。

## ADR-006: 機械学習の「学習」は別 PC、ラズパイは推論のみ

- **状況**: LightGBM の学習（Phase 3）はラズパイ 4B には重く、メモリも食う。
- **決定**: 学習は別 PC（Windows ゲーミング PC 等）で行い、完成モデル（`.pkl`）をラズパイにコピーして推論のみ実行する。
- **理由**: 学習は数か月に 1 回で頻度が低い一方、推論は軽い。ラズパイの資源を推論・配信に集中できる。
- **代替案**: ラズパイで学習まで行う → 時間・メモリ的に非現実的。

## ADR-007: 通知は Discord Webhook（LINE Notify は採用しない）

- **状況**: README 初稿は LINE Notify 前提だった。
- **決定**: 通知の主軸を Discord Webhook にする。
- **理由**: **LINE Notify は 2025 年 3 月 31 日でサービス終了済み**（[jquants.md 出典欄](jquants.md) 参照）。Discord Webhook は無料・ボット登録不要・設定が軽い。LINE が必要なら LINE Messaging API になるが設定が重い。
- **代替案**: LINE Messaging API → 設定コスト高のため後回し。

## ADR-008: J-Quants は Free プランで開発、運用時に有料へ

- **状況**: J-Quants Free は株価が 12 週間遅延（約 2 年分）。短期モメンタムの「今の強い株」は遅延データでは実運用にならない。
- **決定**: 開発・検証は Free で行う。実運用で短期機能を活かす段階で Light（¥1,650）以上に切り替える。
- **理由**: スクリーニング・最適化の**ロジックはプラン非依存**で、違うのは流れてくるデータの鮮度だけ。**DB スキーマもプラン非依存**なので、課金後に作り直しは不要。開発中に課金する必要がない。
- **代替案**: 完全無料に縛る → 短期機能の実運用が成立しないため、運用フェーズでは有料前提とする。

## ADR-009: 日米業種リードラグ戦略は AssetVane の分析機能とする（自動トレードツールに持ち込まない）

- **状況**: 部分空間正則化付き PCA による日米業種リードラグ戦略（SIG-FIN-036, 2026）を機能化したい。別途、自動トレードツールの構想もある。
- **決定**: この戦略は AssetVane の Phase 5 機能（シグナルを提示するだけ）として実装する。自動トレードツールには持ち込まない。
- **理由**:
  - 戦略の本質は「明日どの日本業種を買い/売りすべきかの**日次シグナル**」。シグナル生成と自動発注は別レイヤー。
  - この戦略は日次・低速で、自動トレードが必要とするリアルタイム発注インフラ・tick 監視を一切要求しない。
  - AssetVane の「自動売買せず提示するだけ」という哲学と完全に一致する。
  - 将来無人執行したくなったら、AssetVane が吐いた同じシグナルを執行レイヤーが読めばよく、二重実装は不要。
- **代替案**: 自動トレードツール側で実装 → 日次・低速の戦略に高速執行基盤は不要なため不採用。

## ADR-010: データソースは差し替え可能なアダプタ構成にする

- **状況**: リードラグ戦略（Phase 5）は J-Quants に無い米国 SPDR 業種 ETF の日足を必要とする。
- **決定**: データ取得を共通インターフェースのアダプタ（`JQuantsAdapter` / `UsEtfAdapter` 等）で抽象化する。
- **理由**: 日本株は J-Quants、米国 ETF は Stooq 等と、データ元が複数になる。アダプタ化すれば提供元が変わっても上位ロジックを書き換えずに済む。
- **代替案**: J-Quants 直結のハードコード → 米国 ETF を足せず Phase 5 が詰む。
